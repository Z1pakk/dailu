# 005_orm

## Status
Accepted

## Context
The application needs a data access strategy for persisting domain aggregates to PostgreSQL. Each module has its own bounded context and requires isolation between its persistence concerns. The solution must integrate with clean architecture principles — the domain layer must not depend on persistence details.

Key requirements:
- Domain aggregates must not be polluted with ORM-specific annotations
- Each module owns its own schema and migrations
- Support for complex entity configurations (value objects, owned types, relationships)
- Good integration with the existing .NET ecosystem and PostgreSQL

## Considered Options

1. **Entity Framework Core with Npgsql**
   - Pros:
     - First-class .NET ORM with strong PostgreSQL support via Npgsql
     - Fluent configuration API keeps domain entities annotation-free
     - Code-first migrations per module with separate DbContext instances
     - Built-in support for interceptors (audit, versioning), conventions, and value converters
     - Strong typing, LINQ queries, change tracking
     - Schema isolation via `HasDefaultSchema` per module
   - Cons:
     - Can generate inefficient SQL for complex queries if not careful
     - Learning curve for advanced features (owned types, table splitting)
     - Migrations can become complex in a multi-module setup
   - Use Case: Applications requiring strong typing, complex relationships, and migration management
   - Decision: **Accepted**

2. **Dapper**
   - Pros:
     - Lightweight micro-ORM — full control over SQL
     - Excellent performance for read-heavy workloads
     - Simple to learn and reason about
   - Cons:
     - No change tracking or migration support — must write SQL manually
     - No built-in support for schema evolution
     - More boilerplate for complex mappings
   - Use Case: Read models, reporting queries, performance-critical paths
   - Decision: **Rejected** as the primary ORM — considered for future read-side optimization if needed

3. **Raw ADO.NET**
   - Pros:
     - Maximum control over database interaction
     - No abstraction overhead
   - Cons:
     - Significant boilerplate
     - No migration tooling
     - High maintenance burden for complex domain models
   - Use Case: Extremely performance-sensitive scenarios or legacy compatibility
   - Decision: **Rejected** — too much boilerplate for the benefit gained

## Decision
We use **Entity Framework Core** with the **Npgsql** provider as the primary ORM.

Key conventions and patterns applied across all modules:
- **Separate `DbContext` per module** — each module (Habit, Tag, Identity, etc.) has its own `DbContext` with its own schema, keeping persistence concerns isolated
- **Fluent configuration only** — entity configurations are defined in `IEntityTypeConfiguration<T>` classes; no data annotations on domain entities
- **Snake case naming** via `UseSnakeCaseNamingConvention()` to match PostgreSQL conventions
- **Interceptors** for cross-cutting concerns: `AuditInterceptor` (created/updated timestamps), `VersionInterceptor` (optimistic concurrency)
- **Conventions** registered globally: `DefaultStringLengthConvention`, `SoftDeleteConvention`, `TablePluralizationConvention`
- **Retry on failure** configured for transient PostgreSQL errors (max 3 retries, 30s delay)
- **Migrations per module** stored in each module's Infrastructure project under its own schema history table

## Consequences

**Positive:**
- Domain entities remain clean — no persistence leakage into the domain layer
- Each module's schema is independently versioned and migrated
- Shared interceptors and conventions reduce boilerplate across modules
- Strong LINQ support simplifies most query scenarios

**Negative:**
- Multiple DbContext instances require careful DI registration and connection string management
- EF Core's change tracker can cause subtle bugs if entities are modified outside the tracked context
- Complex aggregates with many value objects require careful EF Core configuration

**Impact:**
- All infrastructure projects reference `SharedInfrastructure` for `AppDbContextBase`, interceptors, and conventions
- Raw SQL or Dapper may be introduced alongside EF Core for complex read queries in the future without changing this decision
